> 系统名称: 宏曦标书 | 版权所有: 云南宏曦科技有限公司 | 版本: v1.2 | 日期: 2026-06-22 | 状态: 已确认

---

## 1. 项目概述

### 1.1 目标

构建一套公司内部使用的 AI 驱动投标书自动生成系统。用户上传招标文件（.docx/.pdf），系统自动解析招标要求，结合公司资源库（资质、人员、历史标书）生成完整的、排版规范的投标文件，经人工审核修订后一键导出 Word + PDF。

### 1.2 当前阶段

- **行业领域**: 保安/物业服务（单一行业，架构预留扩展能力）
- **用户**: 公司投标部门多人使用
- **初期部署**: 云服务器（Docker Compose），架构预留内网迁移能力
- **历史素材**: 初期 < 10 份，系统设计为素材越用越多的飞轮模式，支持持续投喂标书训练系统

### 1.3 核心痛点

传统 AI 工具（豆包等）生成标书的缺陷：
- AI 痕迹重 — 频繁使用"经验丰富""技术精湛"等空泛表达
- 人员配置说明照搬招标文件原文复述
- 缺乏真实公司信息支撑，内容空洞

本系统核心设计理念：**AI 的职责是"组织真实素材 + 扩展专业表达"，而非凭空生成。**

## 3. 核心模块设计

### 3.1 资源库（可复用底座）

系统的核心护城河。AI 生成的每一句话，都尽量锚定到这里的数据。

#### 3.1.1 公司资质库

| 字段 | 说明 |
|------|------|
| 资质名称 | 营业执照、保安服务许可证等 |
| 证书编号 | 用于标书引用 |
| 颁发机构 | |
| 有效期 | 含到期提醒 |
| 附件 | 扫描件 |

#### 3.1.2 人员信息库

| 字段 | 说明 |
|------|------|
| 基本信息 | 姓名、身份证号、学历、联系方式 |
| 从业经历 | 起止时间 + 服务单位 + 担任职务 + 项目规模 + 具体职责 |
| 持证情况 | 保安师证/消防证/安检证等，含证书编号和有效期 |
| 培训记录 | 培训名称、时间、颁发机构 |
| 标签 | 可匹配项（如：工业园区经验、大型活动安保、消防管理） |

人员经历记录需填写**具体化内容**（项目规模数字、具体管理措施、取得成果），避免空泛描述，确保 AI 有"真料"可用。

#### 3.1.3 历史标书库

每次人工定稿后的标书自动入库：
- 按章节片段拆分存入向量数据库
- 关联项目类型、甲方行业、中标结果
- 中标/高分的标书权重提升

### 3.2 招标文件解析模块

- **输入**: .docx / .doc / .pdf（文字型）
- **解析流程**:
  1. 文档提取 — python-docx (Word) / pdfplumber + unstructured (PDF)
  2. 结构识别 — 识别目录、章节、表格、关键段落
  3. AI 信息提取 — 发送解析后的文本给大模型，提取：
     - 项目基本信息（名称、预算、期限）
     - 资质要求
     - 人员配置要求
     - 技术/服务要求
     - 评分标准
     - 特殊要求/否决项
  4. 用户确认 — 以结构化表单展示解析结果，允许人工修正补充

### 3.3 AI 编排 Pipeline（核心引擎）

#### 3.3.1 生成流程

```
Step 1: 需求匹配
  ├─ 招标要求 × 知识库检索 → 匹配可用资质、人员、历史片段
  └─ 输出：匹配报告（哪些要求有现成素材，哪些需要扩展）

Step 2: 标书大纲生成
  ├─ 根据招标文件章节结构 + 行业模板 → 生成大纲
  └─ 用户确认/调整章节结构

Step 3: 逐章生成（核心）
  ├─ 每个章节独立调用 AI，组装 Prompt：
  │   ├─ 系统指令（行业角色 + 写作规范）
  │   ├─ 招标文件相关段落
  │   ├─ 知识库检索到的真实素材（资质、人员、历史案例）
  │   └─ 章节写作要求
  ├─ 向量库检索 → 注入相似历史标书片段作为参考
  └─ 流式返回，前端实时展示

Step 4: 全文润色
  ├─ 连贯性检查
  ├─ 去 AI 痕迹处理
  └─ 排版样式统一
```

#### 3.3.2 防 AI 痕迹策略（5 条约束）

| 策略 | 实现方式 |
|------|---------|
| **空泛词拦截** | 后处理检测"经验丰富""技术精湛""服务周到"等词，统计占比，超标则打回重新生成 |
| **强制引用锚点** | 每段落至少含 1 个具体数字/日期/项目名/证书编号，不满足则追问用户补充素材 |
| **句式多样化** | 检测连续段落开头重复，自动变异句式结构 |
| **关键词精准匹配** | 招标文件的每个"要求"必须在标书中有对应回应段落，且措辞不雷同 |
| **人工审核标记** | 人员姓名、身份证号、证书编号等敏感数据高亮标注，人工确认后才能导出 |

#### 3.3.3 人员章节特化策略

人员章节是 AI 痕迹的重灾区，采用三步生成法：

1. **匹配分析**: 从人员库中按经验标签匹配招标要求，选出最契合的人选
2. **经历扩写**: 将人员履历数据转化为场景化描述。示例：
   - 输入数据: "张三，2018-2022 驻 XX 工业园区，负责安保管理"
   - 输出文字: "张三同志自2018年起驻场XX工业园区担任安保主管，期间主导建立了40人安保团队的轮岗执勤体系。针对园区出入口多、人员流动性大的特点，制定了分时段、分区域的巡逻方案，结合电子巡更系统实现精准管控，驻场期间实现零安全事故。"
3. **语言润色**: 去掉"首先/其次/此外"等模板化连接词，使用行业地道表达

### 3.4 排版渲染引擎

采用"内容 + 样式分离"策略：AI 输出内容结构（JSON），引擎套用排版模板渲染为 .docx。

#### 默认排版规范（可配置）

| 规范项 | 默认值 |
|--------|--------|
| 正文字体 | 宋体 小四（12pt），1.5 倍行距 |
| 一级标题 | 黑体 三号（16pt），段前段后各 0.5 行 |
| 二级标题 | 黑体 四号（14pt）|
| 页边距 | 上/下 2.54cm，左/右 3.17cm |
| 表格 | 表头加粗居中，正文左对齐，实线边框 |
| 封面 | 项目名称居中，投标人 + 日期落款 |
| 目录 | 自动生成，层级可配置 |
| 页眉页脚 | 页眉：公司名称；页脚：第 X 页 / 共 Y 页 |
| 签章区 | 固定留白，方便打印后盖章 |

#### 3.5.2 编辑反馈学习（理解"为什么改"）

这是系统从"好用"到"聪明"的关键。每次人工修订不仅仅是入库一份新标书，更要**让系统理解这次修订的逻辑**。

```
AI 生成版本                    人工修订版本
┌────────────┐                ┌────────────┐
│ 张三同志具  │                │ 张三同志自   │
│ 有丰富的安保 │    Diff 比对   │ 2018年起驻场 │
│ 管理经验，曾 │ ─────────────→ │ XX工业园区， │
│ 参与多个大型 │               │ 期间主导建立 │
│ 项目...     │                │ 40人安保团队 │
└────────────┘                │ 的轮岗执勤   │
                              │ 体系...      │
                              └────────────┘
       │                              │
       └──────────┬───────────────────┘
                  │
                  ▼
         ┌─────────────────────────────┐
         │  AI 分析编辑意图（核心步骤）   │
         │                              │
         │  输入: 原版 + 修订版 + 上下文  │
         │  输出:                        │
         │  {                           │
         │    "edit_type": "具体化",     │
         │    "reason": "原句中'丰富经验' │
         │      太虚，招标方需要看到具体   │
         │      项目、数字、时间",        │
         │    "rule": "人员经历描述必须   │
         │      包含≥2个具体数字(人数/面积│
         │      /时长)，≥1个具体项目名"   │
         │  }                           │
         └─────────────┬───────────────┘
                       │
                       ▼
         ┌─────────────────────────────┐
         │  规则沉淀                     │
         │                              │
         │  每条编辑意图分析结果 →        │
         │  · 是偶发修改？→ 只存范例     │
         │  · 是规律性问题？→ 升级为      │
         │    生成约束规则，注入后续Prompt │
         └─────────────────────────────┘
```

**编辑意图分类**（AI 自动判断）：

| 修改类型 | 含义 | 系统反应 |
|----------|------|---------|
| **具体化不足** | 用户把空泛词换成具体事实 | 提升"强制引用锚点"权重 |
| **风格不对** | 语气、措辞不匹配标书行业习惯 | 注入风格约束到系统 Prompt |
| **信息错误** | 事实性错误（资质号、人名等） | 标记该类信息必须人工确认 |
| **结构调整** | 用户重新编排了章节/段落顺序 | 学习该类型标书的章节偏好 |
| **冗余删除** | 用户删掉了凑篇幅的内容 | 提升"精简性"约束 |
| **补充遗漏** | 用户补充了招标要求中提到但AI未覆盖的点 | 强化招标要点全覆盖检查 |

#### 3.5.3 入库处理流程

每份人工定稿后的标书自动触发入库：

1. **Diff 比对** — 逐段对比 AI 原始输出 vs 人工修订版，标记每一处修改
2. **编辑意图分析** — AI 分析每处修改的"为什么"，输出修改类型和可泛化规则
3. **章节拆分** — 按标书结构（大纲→章→节）自动切分
4. **内容标注** — AI 自动打标签（项目类型、甲方行业、使用的人员、涉及的资质）
5. **向量化** — 每个章节片段生成 embedding 存入向量数据库
6. **关联绑定** — 记录该标书使用了哪些人员、资质、模板
7. **权重计算** — 中标标书权重 ×2.0，累计使用次数高的片段加分
8. **规则升级** — 同类型修改累计 ≥3 次，自动提升为 Prompt 级约束规则

#### 3.5.4 持续进化机制汇总

| 机制 | 做法 | 核心价值 |
|------|------|---------|
| **编辑意图分析** | Diff 比对 + AI 分析每次修改的"为什么"，形成可泛化规则 | 同类问题不再犯第二次 |
| **素材自动入库** | 人工定稿的标书拆成章节片段，存入向量库 | 下次同类项目直接复用 |
| **规则沉淀升级** | 同类型修改累计 ≥3 次，自动升格为 Prompt 级约束 | 系统越改越懂你 |
| **偏好学习** | 追踪用户编辑行为（替换/删除模式），形成用户/公司级偏好画像 | 生成风格越来越贴合 |
| **中标关联** | 中标标书标记高分权重，未中标分析原因 | 生成质量螺旋上升 |
| **增量微调** | 积累足够数据后，基于 DeepSeek 等开源模型 LoRA 微调 | 模型内化公司写作风格 |

#### 3.5.5 初期冷启动策略

初期素材少（< 10 份）时的应对：
- **行业知识注入**: 预置保安/物业服务行业的常见术语、法规引用、服务标准模板
- **Few-shot 示例**: 在 Prompt 中内置 3-5 个高质量的标书写作示例
- **模板先行**: 提供行业标书章节模板（目录结构），减少 AI 对素材的依赖度
- **快速积累期**: 前 20 份标书每份都人工深度打磨，快速建立高质量素材库

### 3.6 消息通知模块

对接企业微信和钉钉，在以下节点自动推送通知：

| 触发事件 | 通知内容 | 渠道 |
|----------|---------|------|
| 标书生成完成 | "XX项目标书已生成，请审核" + 直达链接 | 企微/钉钉 |
| 审核待办提醒 | "XX标书超过24小时未审核" | 企微/钉钉 |
| 投标截止预警 | "XX项目距离投标截止还有3天" | 企微/钉钉 |
| 新素材入库 | "XX项目标书已自动入库，新增X个章节片段" | 站内通知 |
| 系统异常 | AI 调用失败/服务异常告警 | 企微/钉钉 |

实现方式：
- 企业微信：群机器人 Webhook + 应用消息 API
- 钉钉：群机器人 Webhook + 工作通知
- 通知渠道和触发条件可在系统设置中按用户配置开关

## 4. 用户操作流程

### 4.1 首次使用 — 初始化

1. 录入公司资质（营业执照、许可证等）
2. 录入人员信息（含详细从业经历、持证情况）
3. 导入历史标书（如有）
4. 配置默认 AI 模型和排版偏好

### 4.2 每次标书项目

```
┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐    ┌─────────────┐
│ ① 上传招标文件│ → │ ② AI 解析     │ → │ ③ 一键生成   │ → │ ④ 在线编辑   │ → │ ⑤ 导出       │
│              │    │   招标要点    │    │   完整标书   │    │   逐章审核   │    │   Word+PDF  │
│ .docx/.pdf  │    │ 人工确认/补充 │    │ 流式实时展示 │    │ 富文本编辑器 │    │ 排版规范     │
└─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘    └─────────────┘
```

### 4.3 页面结构

- **首页/工作台**: 新建标书项目、继续编辑中的项目、最近项目列表
- **标书项目页**: 核心五步工作流（上传→解析→生成→编辑→导出）
- **资源库管理**: 公司资质、人员信息、历史标书的增删改查
- **系统设置**: AI 模型配置、排版模板、用户管理

## 5. 技术选型

| 层 | 选型 | 理由 |
|----|------|------|
| **前端框架** | React 18 + Vite | 主流，生态好 |
| **UI 组件库** | Ant Design 5 | 企业级后台首选，表单/表格/上传组件成熟 |
| **富文本编辑** | Tiptap | 现代、可扩展，支持实时协作基础 |
| **后端框架** | Python FastAPI | 异步，文档处理生态强，AI/LLM 集成最方便 |
| **ORM** | SQLAlchemy + Alembic | 数据库迁移管理 |
| **业务数据库** | PostgreSQL 15 | 结构化数据（资质、人员、项目、用户） |
| **向量数据库** | Chroma（初期）/ Milvus（规模化） | Chroma 轻量零运维，前期够用 |
| **文件存储** | 本地磁盘（初期）/ MinIO | 初期文件量小，避免运维复杂度 |
| **文档解析** | python-docx + pdfplumber + unstructured | 全覆盖 .docx / .doc / .pdf |
| **Word 生成** | python-docx + docxtpl（模板） | 精确控制排版、样式、封面和目录 |
| **PDF 生成** | LibreOffice headless / python-docx2pdf | Word → PDF 转换 |
| **AI 模型** | Claude API / GPT API / 通义千问（可配置） | 支持多模型切换，避免单点依赖 |
| **任务队列** | Celery + Redis（后期按需引入） | 标书生成较耗时，后台异步执行 |
| **部署** | Docker Compose + Nginx | 一条命令启动，模板化部署 |

## 6. 非功能需求

### 6.1 安全

- 所有外部 AI API 调用走 HTTPS，不传输敏感人员信息（姓名、身份证号等脱敏后再入 Prompt，或仅在本地模板层填充）
- 用户鉴权：JWT Token + RBAC 角色权限
- 数据库访问凭据通过环境变量注入，不硬编码

### 6.2 性能

- 标书生成为长耗时操作（预计 2-5 分钟/份），采用异步任务 + 流式推送（SSE）
- 向量检索目标延迟 < 500ms

### 6.3 可扩展性

- AI 模型通过适配器模式接入，新增模型只需实现统一接口
- 行业模板可配置，新增行业只需添加对应模板 + 知识库
- 排版规范可配置，不同项目可选用不同模板
